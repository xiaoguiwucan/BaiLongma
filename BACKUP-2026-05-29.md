## v0.4.62 - 2026-05-29

### 版本情况
- 版本标签：`v0.4.62`。
- 本版本新增“微信群助手掉线二维码自动通知”，用于解决微信群助手微信号经常掉线后需要手动打开设置页找二维码的问题。

### 更新内容
1. 监控对象是“微信群助手 / 群聊 @ 回复”的微信账号，不是 ClawBot 个人微信账号。
2. 通知通道是“社交媒体 → 微信 ClawBot（个人微信）”，该账号与进群回复的微信群助手账号必须是两个不同账号。
3. 掉线/需要重新登录时，程序会自动生成 Wechaty 登录二维码。
4. 二维码会被渲染成本机生成的 PNG 图片，并发送到 ClawBot 自己；不需要选择联系人、群或接收人。
5. 如果二维码图片发送失败，会降级发送文字和二维码内容，避免完全无通知。
6. 默认 15 分钟重复通知冷却，同一个二维码不会频繁刷屏。
7. 默认开启“离线且暂无二维码时自动重新生成二维码”，避免只提示离线却没有二维码。
8. 设置页“微信群助手”新增掉线通知卡片，可手动开关通知、自动生成二维码和冷却间隔。

### 配置方法
```text
Brain UI → 设置 → 社交媒体 → 微信 ClawBot（个人微信）
先确保 ClawBot 已连接

Brain UI → 设置 → 微信群助手
开启“掉线后自动用 ClawBot 发送登录二维码”
开启“离线且暂无二维码时自动重新生成二维码”
选择重复通知间隔
点击保存并生效
```

### 数据/文件说明
- 二维码 PNG 会生成到本机数据目录：`wechaty-login-qrcode/`。
- 该目录只保存程序自己生成的登录二维码，不会读取或发送用户桌面、相册、截图等本机私有文件。

## v0.4.59 - 2026-05-29

### 版本情况
- 版本标签：`v0.4.59`。
- 本版本是“微信群引用回复可见化”版本，解决引用消息/聊天记录证据已经被程序解析，但群里回复看不出引用依据的问题。

### 问题原因
- 旧版已经能解析微信引用文字、图片、语音、视频、链接、小程序，并把摘要注入 `<wechat-quoted-message>`。
- 但旧提示词只是让模型“需要时短短引用一句”，模型经常直接回答，用户从群里看不到它引用了哪条消息。
- 发送层也没有兜底补引用，所以只要模型省略，最终表现就是“一次引用都没看到”。

### 修复内容
1. 引用消息 prompt 新增 `<wechat-quote-citation-policy>`：依赖引用回答时必须先写一行可见短引用。
2. 聊天记录库检索 prompt 新增 `<wechat-archive-citation-policy>`：使用历史证据时必须显示一条关键聊天记录引用。
3. `send_message` 底层增加自动补引用兜底，模型忘写时仍会发出 `引用 @某某：……`。
4. Wechaty 群消息上下文新增 `raw_payload_text` 和 `message_type`，媒体/XML 引用也能被发送层识别。
5. 新增 `test:wechat-quote-citation`，验证自动补引用和不重复补引用。

### 边界说明
- 本版本不是微信原生引用气泡，不会冒险伪造底层 MsgSource/XML。
- 群里看到的是普通文本引用行，例如：
  - `引用 @风的季节：公益啊`
  - `引用图片 @某某：[图片]`
  - `引用聊天记录：05-29 16:30 风：关键内容`

## v0.4.58 - 2026-05-29

### 版本情况
- 版本标签：`v0.4.58`。
- 本版本是“微信群排行榜身份合并修复”版本，解决同一个群成员被多个历史 sender_id 拆成多个排行榜名次的问题。

### 问题原因
- Web 微信 / `wechaty-puppet-wechat4u` 多次刷新群成员后，同一成员会留下多个历史 `@hash sender_id`。
- 旧排行榜按 `sender_id` 聚合，因此同一个可见昵称会出现在同一个榜单的多个名次。

### 修复内容
1. 排行榜身份合并优先级：`stable_key / wxid / wechat_id` -> 当前群昵称 -> sender_id。
2. 发言榜、发图榜、表情榜、链接榜、装逼榜统一使用新身份合并逻辑。
3. 参与人数统计同步按合并后的身份计算。
4. 每个榜单返回时只保留合并后的成员，不再让同一昵称重复占位。

### 实测结果
- `PT站看片狂魔小群` 今日消息 897，参与 21，榜单无重复昵称。
- `🔥Nas技术之家` 今日消息 426，参与 18，榜单无重复昵称。
- `值班群` 今日消息 75，参与 4，榜单无重复昵称。

## v0.4.57 - 2026-05-29

### 版本情况
- 版本标签：`v0.4.57`。
- 本版本是“微信群图片解析链路修复”版本，重点解决：识图渠道真实可用，但程序后台解析图片总是报 `识图模型返回空内容`。
- 当前本机已重启并加载此版本代码，Electron 监听端口：`127.0.0.1:3721`。

### 问题原因
- 指定识图中转使用原始 `fetch` 请求数据库图片时，可以正常返回 `choices[0].message.content`。
- 旧后台代码使用 OpenAI SDK 调用同一接口，但该中转的响应格式/响应头与 SDK 不完全兼容，SDK 返回被当成字符串对象，导致程序读取不到 `choices[0].message.content`，最终误判为空内容。

### 修复内容
1. 微信图片识别调用改为原始 `fetch + JSON` 解析，和 Skill 页面“测试连通”保持同一套真实多模态调用逻辑。
2. 兼容字符串内容、数组内容、非 2xx 错误体和 Abort 超时，错误信息会写入图片库。
3. 专用 Skill 识图渠道优先于当前 LLM；当前 LLM 只作为兜底，避免聊天模型空返回/超时拖慢后台解析。
4. 同一 Base URL + 模型 + Key 的识图候选去重，不会因为来源是当前模型/Skill/LLM Profile 而重复请求。
5. 开始重新解析时清空旧错误，避免 UI 展示旧错误误导。
6. 超过 15 分钟的陈旧 `running` 图片任务会自动重排为 `pending`，防止程序重启后永久卡住。
7. 状态区改为区分真实待处理、解析中和失败数量；历史坏图不会把整体识图渠道误显示为不可用。

### 实时验证结果
- `/settings/skills/image-vision/status` 返回 `health=ok`。
- 当前默认识图运行时：`skill:gpt-5.4 识图主渠道`。
- 已成功补解析旧卡死图片 id `42`、`48`。
- 已成功解析新图 id `155`、`157`、`158`、`160`、`161`、`165`、`168`、`170`、`171` 等。
- 当前没有真实 `pending/running` 卡死任务；数据库剩余未解析项主要是历史 `error`，原因包括文件/base64 缺失或单张图片超过当前超时。
