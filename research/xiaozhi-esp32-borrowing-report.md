# xiaozhi-esp32 可借鉴能力研究报告

研究日期：2026-05-26  
参考项目：[78/xiaozhi-esp32](https://github.com/78/xiaozhi-esp32)  
研究目标：分析小智为什么反应快、语音是否本地生成，以及白龙马 Mac Electron 助手可以借鉴哪些设计。

## 1. 结论

小智的反应快，不是因为 ESP32 本地跑了完整 AI。它的核心是：

- 本地离线唤醒词；
- 本地 AFE 音频前处理；
- 本地 AEC/NS/VAD；
- Opus 低延迟双向音频流；
- 清晰的设备状态机；
- 服务端流式 ASR + LLM + TTS。

TTS 默认不是 ESP32 本地生成。设备收到服务器下发的 Opus 音频帧后解码播放。也就是说，小智“声音好玩”的部分主要来自服务端 TTS 音色和流式生成能力，而不是设备端本地合成。

对白龙马最值得借鉴的是“实时语音设备架构”，不是直接搬 ESP32 C++ 代码。

## 2. 本地与云端分工

### 小智本地做的事

- 离线唤醒词检测，基于 ESP-SR / WakeNet / AFE。
- AEC 回声消除，可选设备端或服务端模式。
- 噪声抑制、VAD、音频增益等前处理。
- 麦克风 PCM 转 Opus。
- 服务器返回 Opus 音频的解码播放。
- 屏幕状态、表情、提示音、设备电源管理。
- MCP/IoT 设备能力的本地执行。

### 小智服务端做的事

- 流式 ASR。
- LLM 对话。
- TTS 语音合成。
- 声纹识别，文档提到可用 3D-Speaker。
- 会话管理、工具调用、MCP 指令编排。

## 3. 小智为什么快

### 3.1 唤醒词不依赖完整 ASR

小智把“是否被叫醒”和“用户说了什么”拆开。

本地 WakeNet/AFE 先判断有没有唤醒词。只有唤醒后，才打开音频通道进入正式识别。这样避免了完整 ASR 一直处理环境声、视频声和别人说话。

对白龙马的启发：

- 唤醒词要从完整 ASR 中剥离出来。
- ASR 只处理唤醒后的正文。
- 后续应接入真正本地 KWS，例如 sherpa-onnx KWS 或 openWakeWord。

### 3.2 本地 AFE 把音频先处理干净

小智音频前端包含：

- AEC：回声消除；
- NS：噪声抑制；
- VAD：人声活动检测；
- 唤醒词检测；
- 采样率转换和音频缓冲。

对白龙马的启发：

- 麦克风输入不应直接送 ASR。
- 应先做 VAD、AEC、降噪和近场人声判断。
- 视频播放场景尤其需要先做回声消除和视频降音。

### 3.3 Opus 双向流式音频

小智不是录完一整段再上传。它用 Opus 小帧持续传输：

- 设备端上行：16kHz、单声道、Opus，常见 60ms 帧长。
- 服务端下行：Opus 音频帧，设备边收边播。
- JSON 消息负责状态和业务事件，二进制帧负责音频。

对白龙马的启发：

- 语音链路应尽量流式化。
- TTS 不要等整段生成完才播放。
- 前端/后端协议应拆分文本事件和音频帧。

### 3.4 状态机清晰

小智有明确状态：

- Idle；
- Connecting；
- Listening；
- Speaking；
- Activating；
- WifiConfiguring。

状态变化会决定是否开启唤醒词、是否开启语音处理、是否播放提示音、是否允许打断。

对白龙马的启发：

- 需要建立统一语音状态机。
- 不要让前端按钮、ASR 服务、TTS 播放、视频播放器各自管理状态。
- 状态机应输出 UI 状态、日志和调试信息。

## 4. 声纹设计可借鉴点

小智文档提到 3D-Speaker。更重要的是，它在检测唤醒词时保存约 2 秒唤醒词前后音频，并可编码成 Opus 发送给服务器做声纹检测。

对白龙马的启发：

- 增加 pre-roll 环形音频缓存，保留最近 2 到 3 秒麦克风音频。
- 唤醒词音频只用于唤醒和声纹，不进入用户正文。
- 正式 ASR 只识别唤醒后的命令内容。
- 声纹不要只依赖很短的一瞬间，要用更完整、干净的人声段。
- 后续可把 resemblyzer 升级为 3D-Speaker、ECAPA 或 WeSpeaker。

## 5. 视频播放场景启发

用户之前指出的悖论是成立的：如果视频声把用户声音盖住，声纹和 ASR 都会失败。小智的思路是通过 AFE/AEC/VAD 在音频前端尽量解决。

白龙马应采用组合方案：

- 系统 AEC；
- 视频自动降音/暂停；
- 按住说话；
- 本地 VAD；
- 声纹确认；
- 唤醒词阈值动态提高。

这几个功能可以同时开启，不应该互斥。

## 6. 白龙马可借鉴清单

### P0：最应该优先做

- 增加 2 到 3 秒 pre-roll 环形音频缓存。
- 把唤醒词音频、声纹音频、正文 ASR 音频分段。
- 增加语音状态机：Idle、WakeCandidate、SpeakerVerify、Listening、Recognizing、Thinking、Speaking。
- 设置页加入语音调试信息：VAD、音量、声纹分数、唤醒命中、拒绝原因。

### P1：下一阶段做

- 接入真正本地 KWS，替代 ASR 文本唤醒。
- 声纹引擎升级为 3D-Speaker/ECAPA，并支持多样本录入。
- TTS 分句流式播放，减少首句等待。
- ASR partial result 显示和日志记录。

### P2：中长期优化

- 使用 Opus 作为本地/远程音频传输格式。
- 将语音服务协议拆为 JSON 事件和二进制音频帧。
- 加入服务端 AEC 时间戳机制，适配更高级回声消除。
- 将 MCP/工具调用事件与语音/表情/状态分离。

## 7. 不建议直接照搬的部分

- FreeRTOS 任务模型不适合直接搬到 Electron。
- ESP32 AFE/WakeNet 依赖硬件和 ESP-SR，不是 Mac 上直接可用的模块。
- 小智默认依赖官方服务端，不能误认为它是全本地 AI。
- MQTT+UDP 对桌面版不是第一优先级，WebSocket 已经足够。

## 8. 推荐落地路线

### 第一阶段

实现 pre-roll 缓存、语音状态机、调试面板。

### 第二阶段

引入本地 KWS，把唤醒词从完整 ASR 里拆出来。

### 第三阶段

升级声纹模型，支持多样本、多环境录入。

### 第四阶段

改造 TTS 为分句流式播放，支持稳定打断。

### 第五阶段

设计白龙马自己的实时语音协议，向小智的 JSON + Opus 双通道靠拢。

## 9. 对当前白龙马的判断

白龙马现在已经有：

- SenseVoiceSmall；
- Whisper 备用；
- 唤醒词设置；
- 本地声纹；
- 视频降音/暂停；
- 空格按住说话；
- AEC 开关。

下一步真正提升稳定性的关键，不是再堆一个 ASR 模型，而是把音频链路拆清楚：

```text
麦克风输入
→ VAD / AEC / 降噪
→ KWS 唤醒词
→ pre-roll 声纹确认
→ 正文 ASR
→ LLM
→ 流式 TTS
→ 可打断播放
```

这是小智最值得借鉴的地方。
